## ADDED Requirements

### Requirement: Publish product changed event
The system SHALL publish `ProductChangedEvent` after product create, update, on-shelf, off-shelf, delete, or stock-display changes that affect product query results.

#### Scenario: Product update publishes event
- **WHEN** an administrator updates product ID 2001 successfully
- **THEN** the system publishes a `ProductChangedEvent` for product ID 2001 after the transaction commits

### Requirement: Product cache invalidation consumer
The system SHALL consume `ProductChangedEvent` messages and delete affected Redis caches for product detail, hot products, new products, and search results when applicable.

#### Scenario: Product changed event clears caches
- **WHEN** the cache consumer receives a `ProductChangedEvent` for product ID 2001
- **THEN** the system deletes the product detail cache and related product listing caches

### Requirement: Product cache key contract
The system SHALL centralize product cache key creation and deletion behind a `ProductCacheService` or equivalent helper. The helper SHALL own keys for product detail, hot products, new products, and product search cache so writers and invalidators use the same key contract.

#### Scenario: Product cache service owns cache keys
- **WHEN** product detail cache is written and later invalidated for product ID 2001
- **THEN** both operations use the same product detail cache key generated by the shared cache helper

### Requirement: Cache invalidation is idempotent
The system SHALL treat repeated cache invalidation events as successful when the target cache keys are already absent.

#### Scenario: Duplicate cache invalidation is safe
- **WHEN** the same `ProductChangedEvent` is consumed twice
- **THEN** the second consumption completes successfully without recreating or corrupting cache data
